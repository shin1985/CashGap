1. **会計年度開始月の変更が「ラベル変更だけ」になっている**

   * `months` は `getFiscalMonthLabels(params.fiscalYearStart)` で作っていますが、`monthly` 配列自体を回す処理がありません（`src/App.jsx:123`, `1287-1291`）。
   * 影響: 4月に入れていた値が、開始月を1月に変えるとそのまま1月の値として表示されます。
   * これはかなり重大で、**既存データの月の意味が変わってしまう**実装です。

2. **全行削除した状態を保存できない。再読込でサンプルが復活する**

   * `loadWorkbook()` で配列が空だと `defaultState` に戻しています（`src/lib/sheetStore.js:288-291`）。
   * 影響: たとえばプロジェクトを全部削除して保存しても、再読込するとサンプル案件が戻ります。
     `bankManualIncomeRows` / `bankExpenseRows` / `plRows` も同様です。
   * これは**データ消失というより“意図しないデータ復活”**なので、かなり危険です。

3. **保存処理が破壊的で、途中失敗時にシートを空にしうる**

   * 保存は `batchClear` → `batchUpdate` の2段階です（`src/lib/sheetStore.js:201-224`）。
   * 影響: `clear` 成功後に通信失敗すると、**ワークブックだけ空になって更新されない**可能性があります。
   * 保存系としては最優先で直したい箇所です。

4. **`0` を正しく保持できない箇所がある**

   * `applySnapshot()` で `Number(...) || DEFAULT_PARAMS...` を使っているため、`0` が既定値に置き換わります（`src/App.jsx:131-134`）。
   * freee 取込でも `startingCash: bs.bsEstimates.startingCash || prev.startingCash` になっていて、`0` が前回値に負けます（`src/App.jsx:392-395`）。
   * 影響: `0円`, `税率0%`, `回収日数0日` などの正しい値が保存・読込後に壊れます。

5. **freee 取込後、Projects → PL の自動連動が壊れる**

   * freee 取込時に `setPlRows(pl.plRows)` で丸ごと置換しています（`src/App.jsx:388-390`）。
   * しかも parser 側が作る行は `autoLink: false` です（`src/lib/freeeParser.js:198-205`, `231-237`）。
   * 影響: 取込後は Projects タブで売上を直しても、PL の売上高に反映されなくなります。
   * アプリの中核機能とぶつかっています。

6. **freee CSV の期間情報を使っていないので、月ズレが起きうる**

   * `periodFrom` / `periodTo` は parse しています（`src/lib/freeeParser.js:72-88`）。
   * でも import 時に `fiscalYearStart` へ反映していません（`src/App.jsx:383-400`）。
   * 影響: CSV が 1月始まり、画面が 4月始まり、のようなケースで**列の意味がずれる**可能性があります。

7. **BS CSV を読んでも、実際に反映しているのはほぼ期首現金だけ**

   * parser は `receivableOpening` / `payableOpening` まで出しています（`src/lib/freeeParser.js:271-317`）。
   * でも import 時には `startingCash` しか state に反映していません（`src/App.jsx:391-395`）。
   * 影響: 「BS をインポート」と見えても、実際は**かなり限定的な取込**です。

8. **税金の計算が月次黒字ベースなので、年次では過大計上になりやすい**

   * `Math.max(0, 月次経常利益 * taxRate)` で月ごとに税金を出しています（`src/lib/calculations.js:107-108`）。
   * 例: 4月 +100、5月 -100 なら年合計 0 でも、アプリ上は 4月分だけ課税されます。
   * 年度PLとして見るなら、かなり不自然です。

9. **未保存変更を失いやすい**

   * `handleOpenWorkbook`, `handleReloadWorkbook`, `handleResetSamples` に `isDirty` の確認がありません（`src/App.jsx:222-275`）。
   * 影響: 編集途中で別 workbook を開く、再読込する、サンプルへ戻す、で**そのまま消える**可能性があります。

10. **freee パーサの堅牢性が弱い**

* パース失敗時に、前回成功した結果を state から消していません（`src/App.jsx:373-378`）。
* CSV parser が「1行=1レコード」前提で、**引用符内改行**を扱えません（`src/lib/freeeParser.js:22-68`）。
* BS の現金判定がキーワード依存なので、勘定科目名によっては期首現金を取りこぼします（`src/lib/freeeParser.js:259-292`）。

11. **README と実装に不整合がある**

* README は `.env.example` をコピーしろと書いていますが、リポジトリにそのファイルがありません（`README.md:41-47`）。
* README は freee import を Settings タブと書いていますが、実装は Storage 画面です（`README.md:72`, `src/App.jsx:504-573`）。


修正優先度を付けるなら、この順がよいです。

3. 保存処理の破壊性
4. `0` の扱い
